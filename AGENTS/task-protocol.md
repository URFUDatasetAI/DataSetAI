# Task Protocol

Этот файл задаёт правила для `AGENTS/tasks/*`.

## Когда создавать task-file

Создавай новый task-file, если выполняется хотя бы одно условие:

- работа явно не закончится одной правкой;
- затрагиваются несколько подсистем или несколько инвариантов;
- уже появились решения, которые следующий агент легко потеряет без handoff;
- пользователь явно просит продолжить тему позже;
- в ветке появляется незавершённое состояние, которое нельзя безопасно “догадать” по diff.

Не создавай task-file для маленьких локальных фиксов, которые полностью закрываются в одном изменении и не оставляют хвоста.

## Когда обновлять task-file

Обновляй task-file:

- перед крупной серией правок, если план изменился;
- после заметного прогресса;
- когда принят новый компромисс или обнаружен риск;
- когда меняется статус задачи;
- перед handoff другому агенту.

## Что должно быть в task-file

Минимальный состав:

- `Status`
- `Why it exists`
- `Scope / non-goals`
- `Relevant files`
- `Decisions already made`
- `Open edges / risks`
- `Next likely steps`
- `Handoff notes`

Пиши не только “что меняли”, но и “зачем это важно” и “что нельзя потерять”.

## Status Vocabulary

Используй короткие статусы:

- `active`
- `blocked`
- `monitoring`
- `ready`

Если тема завершена полностью, task-file обычно удаляется из `tasks/`, а итог при необходимости кратко переносится в [current-state.md](current-state.md).

## Linking Rules

- Если задача родилась из backlog-идеи, добавь ссылку на источник в `ideas/*`.
- Если во время задачи становится понятен новый устойчивый архитектурный компромисс, обнови релевантный файл в `context/*`.
- Если задача меняет картину проекта прямо сейчас, отрази это в [current-state.md](current-state.md).
- Не создавай записи в `ideas/*` только потому, что в документации видны возможные future extensions. Для этого репозитория ideas-layer заполняется только после явного обсуждения и подтверждения, что идею вообще нужно учитывать как backlog.

## Handoff Minimum

Нельзя оставлять после себя “немой” незавершённый diff. Перед handoff зафиксируй:

- что уже работает;
- что ещё не работает или не проверено;
- какие файлы были центральными;
- какой следующий шаг наиболее вероятен;
- какие инварианты особенно легко случайно сломать именно здесь.

## Cleanup Rule

`tasks/` - это слой живой незавершённой работы, а не архив всех когда-либо сделанных задач. Если тема закрыта и не требует сопровождения, убери её отсюда.
